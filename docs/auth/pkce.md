# PKCE Notes

## What PKCE is

PKCE stands for Proof Key for Code Exchange.

It is an OAuth 2.0 extension that protects authorization code flows for public clients, especially:

- browser apps
- mobile apps
- desktop apps
- any app that cannot safely keep a client secret private

Without PKCE, a stolen authorization code can sometimes be exchanged by an attacker. PKCE reduces that risk by binding the authorization code to a one-time secret generated by the client.

## Why it exists

Traditional OAuth authorization code flow assumes the client can safely store a client secret.

That works for backend services, but not for browser apps or mobile apps, because users can inspect shipped code and network traffic. In those environments, a secret is not really secret.

PKCE solves that by replacing the trust in a static client secret with proof that the same client that started the flow is the one finishing it.

## Where PKCE is used

PKCE is commonly used in:

- single-page apps
- browser apps
- mobile apps
- desktop apps

It is now considered best practice even beyond those contexts, but it is especially important for public clients.

## How PKCE works

High-level flow:

1. The client generates a random `code_verifier`
2. The client derives a `code_challenge` from that verifier, usually with SHA-256
3. The client sends the user to the authorization URL with the `code_challenge`
4. The authorization server authenticates the user and returns an authorization `code`
5. The client sends that `code` plus the original `code_verifier` to the token endpoint
6. The server recomputes the challenge and verifies it matches
7. If it matches, the server returns tokens

Important pieces:

- `code_verifier`: random high-entropy string kept by the client
- `code_challenge`: derived value sent up front
- `code_challenge_method`: usually `S256`

The most important point is this: the browser never proves identity with a long-lived secret. It proves continuity with the verifier it generated at the start of the flow.

## How this applies to Spotify

For Spotify, PKCE is the right fit for browser-based examples and real browser apps.

Spotify supports:

- authorization code flow for confidential clients
- authorization code with PKCE for public/browser clients
- client credentials for server-to-server catalog access

In practice:

- if the app is server-only and can safely hold `clientSecret`, normal auth code flow is acceptable
- if the app runs in the browser, PKCE is the safer flow
- client credentials are only for app-level access, not user data

That means for `spotify-effect`:

- `#11` covers server-to-server app tokens
- `#12` should cover browser-safe user auth via PKCE and server-style user auth via auth code + refresh token

## What we need to build for Spotify PKCE

For the library, the practical pieces are:

- generate authorization URLs with PKCE parameters
- generate or accept a `code_verifier`
- exchange the returned code with that verifier
- return refreshable user tokens
- refresh the access token automatically when the token provider knows it is expired or missing

For the browser example, the practical pieces are:

- generate a verifier and challenge
- redirect to Spotify auth
- handle the callback code
- exchange the code for tokens through the library
- store only what the example needs for the session

## Suggested mental model

Use this shortcut:

- client credentials = app auth, no user
- auth code = user auth for confidential/server clients
- PKCE = user auth for public/browser clients

## Recommended implementation default for this repo

For `#12`, automatic refresh should happen only when the token provider knows the token is missing or expired.

Keep the first implementation:

- in-memory
- simple
- explicit about when expiry is checked

Do not overbuild persistence, session storage, or browser token management until the base flow is working cleanly.
